# 申請ガイド生成フロー（正本）

このファイルは、役員会議で「採用」と決まった案件を、外部委託先（社労士・行政書士）に渡せる状態まで持っていくためのフローを定義する正本です。
ChatGPT Pro（設計監査役）のレビューを反映して確定。

---

## 設計思想

1. **LLMには申請書を書かせない** — ハルシネーションリスクが高い。代わりに「質問・証拠・段取り」を作らせる
2. **「内部で素材を揃える → 外部で清書」を分担の基本**にする
3. **属人化を防ぐ** — トリガー、プロンプト、フォルダ構成、すべて標準化する
4. **採用決定前に Go/No-Go を判定する** — リードタイム不足で見切り発車させない
5. **case.md を単一の状態管理ソースにする** — 状態遷移をトリガーに定型処理が走る

---

## LLM生成の境界線

### 生成する（LLMが価値を出せる領域）

- 課題→施策→効果→KPIの骨子
- 社内向け質問票（欠けている数字・証憑の洗い出し）
- claim-evidence 対応表
- KPI候補のメニュー
- 申請理由の叩き台（役員会議の決定理由を文章化）
- 制度要件の要約（公式資料の引用付きに限る）

### 生成しない（ハルシネーション危険）

- 最終版の申請書本文
- 適格性の断定
- 補助対象経費の断定
- KPIの実数値（baseline は社内実測のみ信頼）
- 効果額の断定

## フロー全体

```
[役員会議で「採用決定」]
   ↓
[case.md の status を 採用決定 に更新]
   ↓
[人間が prompts/scaffold-application.md をClaude Codeに投げる]
   ↓
[Claude Code]
  ├─ 制度名・公式URL・case_id を受け取る
  ├─ data/adopted/{case_id}/ フォルダを作成
  ├─ 公式資料を evidence/official/ に保存（PDFスナップショット）
  ├─ 6ファイルを scaffold 生成（事実は人間が埋める前提）
  └─ case.md にownerなどガバナンス情報を追記
   ↓
[00_preflight.md を人間が埋めてGo/No-Go判定]
   ↓ Go の場合
[02_materials_pack.md の質問を埋めながら evidence/company/ を集める]
   ↓
[03_external_brief.md を完成させて外部委託先へ送付]
   ↓
[外部委託先が清書 → 申請（submitter が提出ボタンを押す）]
   ↓
[04_submission_log.md に提出履歴を記録]
   ↓
[case.md のstatusを 申請済 に更新 → 以降のステータス遷移に従う]
```

## 6ファイルの責務

### case.md
状態管理の単一ソース。詳細スキーマは `doc/data-schema.md` を参照。

### 00_preflight.md（事前適格性チェック）

採用決定後 **24時間以内** に埋める。Go/No-Go判定を出すためのファイル。

含めるべき項目:

- 制度名、公募回、締切
- 提出システム（JGrants / 独自ポータル / 助成金ポータル / e-Gov / その他）
- GビズID取得状況（プライム取得済か / メンバー作成済か）
- 必須前提条件のチェック（SECURITY ACTION、認定支援機関、ITツール登録など）
- 施設・実施場所・使用権の確認
- 他補助金との重複・実施中制限のチェック
- 相見積取得可否
- baseline 数値の取得可否
- 外部委託先の着手枠
- **Go / No-Go 判定**（赤項目があれば No-Go、次回公募に向けて素材前倒し）

### 01_runbook.md（制度要点と段取り）

実務で参照される作業書。長い読み物にしない。

含めるべき項目:

- 制度の要点（補助率・上限・対象経費・特殊要件）
- 公式資料一覧（evidence/official/ へのリンク）
- 役割分担（内部 / 外部 / submitter）
- マイルストーン（申請締切から逆算）
- 必要書類一覧
- 落とし穴（prompts/traps.md から該当項目を引用）
- 差戻し時の連絡先

### 02_materials_pack.md（素材パック）

**LLMが最も価値を出すファイル**。文章草案集ではなく、質問と証拠の束。

含めるべき項目:

- 事業の目的
- 現状課題
- 実施内容
- 期待効果
- KPI候補（メニュー形式・人間が選ぶ）
- **claim-evidence matrix**（主張ごとに、それを裏付ける証拠ファイルを対応付け）
- 未回答質問リスト（社内に投げる質問）
- 会社固有情報の埋込欄

### 03_external_brief.md（外部引継ぎ仕様書）

依頼文ではなく **handover spec**。外部委託先が最短で動けるための情報。

含めるべき項目:

- 制度名・公募回・締切・公式資料版
- 提出システム
- 誰が最終提出するか（submitter）
- 対象施設・事業実施場所
- 会社の意思決定理由
- 実施内容・予算・見積状況
- baseline と KPI
- 未解決論点
- 依頼範囲（清書だけか、要件確認も含むか）
- 成果物・期日
- 連絡窓口
- 費用条件（成功報酬の有無等）

### 04_submission_log.md（提出履歴）

提出後の追跡用。

含めるべき項目:

- システム名
- アカウント名義
- 提出担当
- 下書き作成日 / 最終提出日
- 申請番号 / 受付番号
- 差戻し履歴
- 追加提出物
- 最終提出版ファイル一覧

### evidence/

実物ファイル置き場。

- `evidence/official/` — 公募要領・FAQ・様式・スケジュール（PDFスナップショット）。後で「どの版で書いたか」を追えるようにする
- `evidence/company/` — 決算書・登記・見積・KPI実測値・施設情報

## 制度4類型と必要リードタイム

採用決定から申請までの所要時間は制度によって大きく異なる。3週間前通知に対応できない類型もある。

| 類型 | 例 | 現実的な所要 | 3週間前通知への対応 |
|---|---|---|---|
| 雇用系助成金 | キャリアアップ、人材開発、業務改善 | 2〜4週間 | 条件揃えば可 |
| 登録事業者連携型 | IT導入補助金 | 3〜4週間 | アカウント・支援事業者揃えば可 |
| 設備投資・中規模計画型 | ものづくり、省力化（一般型）、観光庁設備系 | 5〜8週間 | **厳しい** |
| 大型事業計画型 | 新事業進出補助金 | 6〜8週間以上 | **原則見送り** |

**3週間前通知に当たった場合**: 24時間以内に preflight を回し、赤項目があれば **その回は見送り、次回公募に向けて素材を前倒しで作る運用** に切り替える。

## トリガー設計

### 当面（手動）

1. 役員会議で採用決定
2. case.md を新規作成（status: 採用決定）
3. 人間が `prompts/scaffold-application.md` を Claude Code に投げる
4. Claude Code が scaffold 生成

### 将来（自動化候補）

case.md の status が「採用決定」に変わったことを GitHub Actions が検知 → Claude Codeを呼び出す形にできる。ただし当面は手動で十分。

## よくある失敗パターンと防ぎ方

| 失敗 | 何が起きるか | 防ぎ方 |
|---|---|---|
| 内部が抽象的な文章だけ渡す | 外部がゼロから書き直す | claim-evidence表を先に作る |
| 施設や実施場所が未確定 | 外部が要件判定できない | preflightで対象施設を固定する |
| 見積条件がバラバラ | 経費整理がやり直し | 同一条件の見積依頼テンプレを使う |
| KPIの数字がない | 効果欄が空疎になる | baseline責任者を決める |
| 誰が提出するか曖昧 | 締切日に止まる | submitter を case.md で固定する |
| 外部のスコープが曖昧 | 「どこまでやるか」で揉める | external_brief に成果物・期日・範囲を明記 |
