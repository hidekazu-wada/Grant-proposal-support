# アカウント運用ルール（正本）

このファイルは、補助金申請に必要な各種アカウント（特にGビズID）の運用ルールを定義する正本です。
担当者の退職・不在時の継続性を確保するための仕組みを定めています。

---

## なぜこのルールが必要か

- GビズIDは **共用禁止**（共用するとアカウント停止リスク）
- 担当者が1人だけだと、退職・休職・不在時にすべての申請が止まる
- JGrants は補助金ごとに代理申請可否が異なるため、誰がどのアカウントで提出するかを事前に決めておく必要がある
- 2026年7月以降、GビズIDに有効期限管理が入る予定

---

## GビズID運用ルール

### 原則

1. **代表者プライムID** は会社の正式アカウント。日常運用には使わない
2. **メンバーアカウント** を担当者ごとに発行する（即日発行可）
3. アカウントを **複数人で共用してはならない**
4. 担当者が変わるたびに、メンバーアカウントを **追加・削除** する

### アカウント体系

| 種別 | 用途 | 持つ人 |
|---|---|---|
| プライム（代表者） | 会社の正式アカウント。役職者の交代時のみ更新 | 代表取締役 |
| メンバー（主担当） | 日常の補助金申請業務 | 申請主担当者 |
| メンバー（副担当） | 主担当不在時のバックアップ | 申請副担当者 |
| メンバー（外部委託先用） | 必要に応じて発行（代理申請可の制度のみ） | 社労士・行政書士 |

### 通知メール設定

JGrants は **追加通知先メールアドレスを最大10件** 設定可能。

- 主担当のメール
- 副担当のメール
- 共有メーリングリスト（管理部全員に届く）

最低この3つを設定し、**1人不在でも差戻しを見落とさない** 体制にする。

## case.md でのアカウント記録

各案件の case.md には、以下を必ず記録する:

```yaml
governance:
  owner:           # 主担当
  backup_owner:    # 副担当（必須）
  submitter:       # 実際に申請ボタンを押す人
  portal:          # JGrants / 独自ポータル / 助成金ポータル / e-Gov
  gbizid_role:     # プライム / メンバー
```

## 制度ごとの代理申請可否（参考）

| 制度 | 代理申請 | 注意点 |
|---|---|---|
| ものづくり補助金 | 不可（本人申請のみ） | 代理機能が無いため社内で submitter を確保 |
| 中小企業省力化投資補助金（一般型） | 応募申請: 本人 / 交付申請: 代理可 | 段階で運用を切り替える |
| IT導入補助金 | 支援事業者が作成支援、提出は申請者 | IT導入支援事業者と契約必要 |
| 新事業進出補助金 | 申請者自身で事業計画作成 | 外部支援OKだが本人提出 |
| 雇用関係助成金 | 代理人電子申請可 | 社労士に代理権限を付与 |
| 観光庁 観光地・観光産業における省力化投資補助 | 計画申請締切の1週間前にアカウント発行期限 | 早めの参加申込必須 |

## 必要なその他のアカウント・登録

| 項目 | 用途 | 取得目安 |
|---|---|---|
| GビズIDプライム | JGrants 利用の前提 | 書類郵送で受領から1週間 |
| GビズIDメンバー | プライム配下で発行 | 即日 |
| SECURITY ACTION 宣言 | IT導入補助金の必須要件 | 即日〜数日 |
| 独自ポータル各種 | 制度ごとに別アカウント | 制度により異なる |

---

## 担当者交代時の手順

1. 新担当者のGビズIDメンバーアカウントを発行
2. JGrantsの追加通知先メールに新担当者を追加
3. 旧担当者のメンバーアカウントを停止（退職時は即日）
4. 進行中の case.md の owner / backup_owner を更新
5. 引き継ぎ時に `data/adopted/{case_id}/` 配下を全部レビュー
