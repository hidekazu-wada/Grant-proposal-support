# Grant Proposal Support（助成金キャッチアップBOT）

## プロジェクト概要

株式会社HAMAYOUリゾートが活用できる助成金・補助金情報を、自分から取りに行かなくても定期的に手元に届く仕組みを作る。会議で「使う／見送る」を判断できる粒度で情報を整え、採用したものについては申請手順までナビゲートし、活用中の助成金とその効果を数値で可視化する。

**目的:**

- 「知らなかったから使えなかった」をなくす
- 助成金の検討〜申請〜効果測定までを1つの流れにする
- 情報収集の属人化を解消し、会議のアジェンダ化できる形にする

**現在のフェーズ:** フェーズ1 — 実装・運用準備

**制約・前提:**

- UIはChatWorkに集約する（既に社内で使われているため）
- 月1回の通知ペースをベースラインとする
- 機密データの扱いは現時点では大きな問題にならない見込みだが、将来出てくる可能性があるためアーキテクチャ上は外部LLMに渡す情報を絞れる構造にする
- 会社の業種・施設構成は `~/.claude/projects/.../memory/company_hamayou.md` を参照

## システム全体の流れ

```
[月1トリガー: 毎月第1金曜 朝7時（祝日なら前日）/ Perplexity Computer]
   ↓
[Perplexity Computer]
  ├─ HAMAYOUリゾートの会社プロファイルを前提に助成金リサーチ
  ├─ プロンプト/出力テンプレ（このリポジトリ prompts/ が正本）に従って整形
  └─ data/grants/YYYY-MM.md として本リポジトリに push（GitHub公式コネクタ経由）
   ↓
[GitHub Actions: push検知 or 月次cron]
  └─ 当月の data/grants/YYYY-MM.md を読み、ChatWork API にPOST
   ↓
[ChatWork通知] 会議で読める粒度のメッセージ
   ↓
[気になる候補を ChatGPT Pro に投げてメリット/デメリット/適合度/申請難易度を分析]  ← 会議の前
   ↓
[会議で判断] 採用 / 見送り / 追加調査
   ↓
[採用したもの] 申請手順ガイド生成 → data/adopted/ に追記
   ↓
[活用ログ] 申請額・採択額・入金額・工数を記録 → 月次サマリ
```

**設計上の分離:**

- **収集レイヤー**（Perplexity Computer）と **通知レイヤー**（GitHub Actions + ChatWork API）と **判断支援レイヤー**（ChatGPT Pro）を分ける
- **データの正本** はこのリポジトリ側（Git）に持つ。Perplexity・ChatGPT のワークスペースは作業場であり正本ではない
- **会議の前にChatGPT Proを通す** ことで、会議は「ゼロから議論する場」ではなく「判断する場」になる

## 会議体との接続

| 項目 | 内容 |
|---|---|
| 連動する会議 | HAMAYOUリゾート 役員会議（毎週金曜） |
| 参加者 | 取締役・執行役員（必要時に現場管理職） |
| 通知タイミング | **毎月 第1金曜日 朝7時**（第1金曜が祝日の場合は前日に前倒し） |
| 議題化 | 第1金曜日の役員会議の **冒頭議題** に固定 |
| 読み方 | 事前読み込みは期待しない。**会議の場で読み上げて議論する** 前提で通知文を設計する |
| 即決ルール | 内容次第で速決。判断が割れた候補は **翌週金曜の役員会議** に持ち越す |
| 持ち越し時のサポート | 必要に応じて ChatGPT Pro（助成金アドバイザー）でメリデメ・適合度・申請難易度を取得してから次回会議へ |
| 速報通知 | 現状なし。月次のみで運用 |
| 投稿順 | **個別通知（1助成金1メッセージ） → 最後にサマリ通知** の順で投稿。サマリから個別にリンクする |
| 会議後の記録 | 決定事項は個別メッセージへの **返信** として【会議決定】定型で残す |
| 会議後のタスク化 | 「採用」「条件付き採用」になった案件は、その場で **ChatWorkタスク化**（担当・期限付き） |
| 投稿アカウント | **BOT専用ChatWorkアカウント**（または `self_unread`）を使い、自分宛て投稿が未読にならない問題を回避 |

- LLM間のデータ受け渡しは人間 or Git経由

## 成功指標・撤退条件

詳細な指標定義と算式は `doc/data-schema.md` を参照。
ここでは判定軸のみ示す。

### 成功指標

#### A. BOT/仕組みの評価（ファネル指標）

| 指標 | 目標値 | 算式 |
|---|---|---|
| 候補件数 | 月平均1件以上 | discovered_count |
| 議題化率 | 50%以上 | agenda_count / discovered_count |
| 採用率 | 30%以上 | adopted_count / agenda_count |
| 申請化率 | 75%以上 | applied_count / adopted_count |
| 入金化率 | 80%以上 | paid_count / applied_count |
| BOT起点率 | 50%以上 | bot_origin_cases / adopted_count |

#### B. 個別案件の価値

| 指標 | 目標 | 算式 |
|---|---|---|
| 申請純便益 | プラス | paid_amount - external_cost - internal_cost_est |
| 実質補助率 | 30%以上 | paid_amount / actual_total_cost |
| KPI改善量 | baseline比で改善 | actual - baseline |

### 撤退条件（ファネル分解で原因を特定）

| 状況 | まず疑うべき点 | アクション |
|---|---|---|
| 候補ゼロ + 議題ゼロ | 情報源 / BOT設計 | prompts/sources.md とリサーチプロンプトを見直し |
| 候補あり + 議題ゼロ | ChatWork通知 or 会議運用 | 通知文フォーマットと会議導線を見直し |
| 議題あり + 採用ゼロ | 対象会社との適合性 | prompts/traps.md を強化、フィルタを厳しく |
| 採用あり + 申請ゼロ | 社内工数 / 申請体制 | 外部委託の検討、申請フローの整備 |
| 申請あり + 採択ゼロ | 要件判定 or 申請品質 | 採択率の高い制度へ軸足移動 |
| 採択あり + 入金ゼロ | 実施 / 実績報告 / 証憑対応 | 証憑管理の仕組みを整備 |

| 判定 | 閾値 | アクション |
|---|---|---|
| ソフトストップ | 6ヶ月連続で議題化ゼロ | 上表に従って原因特定 → 該当箇所を改善 |
| ハードストップ | 1年運用して上表のどこにも改善余地が見えない | プロジェクト終了の再検討 |

## 重要な設計判断の履歴

| 日付       | 判断                                                                                         | 理由                                                                                       |
| ---------- | -------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------ |
| 2026-04-09 | UIをChatWorkに集約                                                                           | 既に社内で使用中。新しいUIを増やさない                                                     |
| 2026-04-09 | 収集（Perplexity Computer）と通知（GitHub Actions + ChatWork API）と判断支援（ChatGPT Pro）の3層分離 | 各層を差し替え可能にし、データの正本をGitに集約するため                                    |
| 2026-04-09 | Perplexity Computer から ChatWork へは GitHub経由（直接投稿しない）                          | ChatWork公式コネクタが無く、MCPサーバー自作は月1の用途に対して重い。GitHub公式コネクタで代替 |
| 2026-04-09 | ChatGPT Pro を「設計監査役」と「助成金アドバイザー」の2役で採用                              | 会議の前に候補のメリデメを把握しておくことで、会議を判断の場にできる                       |
| 2026-04-09 | テンプレ由来の「パラメータ凍結」「検証パイプライン」「3層記録フォーマット」を削除            | FX自動売買向けの構造であり、本BOTプロジェクトには馴染まない                                |
| 2026-04-09 | 効果測定の管理単位を「1助成金」ではなく「1申請案件」に。施設×年度×公募回で識別 | 同じ制度でも施設ごとに別申請になるため、混ぜると効果測定不能（ChatGPT Pro レビュー反映） |
| 2026-04-09 | 金額を申請額／交付決定額／確定額／入金額の4段階で記録                                       | 補助金は採択額と入金額がズレる。4段階で記録しないと翌年の見積もり精度が出ない              |
| 2026-04-09 | ステータスから「完了」を廃止し「義務終了」を導入                                            | 入金後3〜5年の事後報告義務を追わないと返還リスクを見落とす                                  |
| 2026-04-09 | 工数を6つの内訳（screening/meeting/application/vendor_coordination/result_report/post_report）で記録 | 申請より後工程の方が重い案件が多く、合計値だけだと判断材料にならない                       |
| 2026-04-09 | 軸C（事業インパクト）を任意ではなく定量KPI最低1つを必須に                                   | 任意項目は誰も書かない。採用決定時にbaselineを取らせる強制力が必要                          |
| 2026-04-09 | data/pipeline/ にBOT母集団ログを別ファイルで持つ                                            | 採用ゼロの年に「BOTが悪い／通知が悪い／適合性が悪い」のどれかを判別できなくなるため         |
| 2026-04-09 | 通知を毎月第1金曜朝7時（祝日なら前日）に固定し、その日の役員会議の冒頭議題にする | 役員会議が毎週金曜開催・事前読み込みしない文化のため、当日朝に出してその場で議論するのが最適 |
| 2026-04-09 | 申請ガイドはLLMに「申請書」を書かせず「質問・証拠・段取り」を作らせる方針に確定 | ハルシネーションリスクを排除しつつ、内部素材整理にLLMの価値を最大化（ChatGPT Pro レビュー反映） |
| 2026-04-09 | data/adopted/ を1案件1フォルダ構成に変更し、6ファイル + evidence/ サブフォルダで管理 | 公式資料スナップショットと社内資料を物理分離し、属人化と「どの版で書いたか不明」を防ぐ |
| 2026-04-09 | アカウント運用ルール（GビズIDプライム+メンバー、共有禁止、共有通知メール）を doc/account-governance.md で正本化 | 担当者退職リスクへの備えと、JGrants共用禁止への対応 |
| 2026-04-10 | ChatWork通知を「会議用判断カード + 議事進行サマリ」の2種構成で確定（prompts/chatwork-format.md） | 役員会議当日朝に読み上げて議論する前提。単一スコアではなく2軸評価+仮判断ラベル、投稿順は個別→サマリ、会議後は返信+タスク化で記録（ChatGPT Pro レビュー反映） |
